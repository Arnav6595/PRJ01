1. Overview

This document covers the complete quality-engineering strategy and implementation outcome for the Wipro Playwright Batch 2 Capstone Project.

The Application Under Test (AUT) is Practice Software Testing — Toolshop (practicesoftwaretesting.com), a fully functional Angular-based e-commerce platform simulating real-world retail operations. The system supports a complete customer journey: product discovery, search and filtering, shopping-cart management, multi-step checkout with several payment methods, customer support, invoice/order history, and a privileged administrative dashboard.

The goal was to design and implement a production-ready Playwright automation framework from scratch in JavaScript that:

  • Isolates front-end UI testing from a volatile, third-party backend.
  • Handles an Angular Single-Page Application reliably using auto-waiting locators and response-driven waits.
  • Runs unattended in a Continuous Integration pipeline with rich diagnostic reporting.
  • Demonstrates advanced techniques: network mocking, accessibility auditing, and visual integrity checkpoints.

The engagement followed the complete Software Testing Life Cycle (STLC) — requirement analysis, test strategy, framework construction, execution, defect documentation, and reporting. The final automated suite contains 139 tests across 11 spec files, executing in parallel across a three-way CI shard matrix at a 100% pass rate.

2. Goals

2.1 Business Objectives

  • Transaction Integrity: guarantee correctness of the end-to-end checkout flow — address entry, payment-method selection (Cash on Delivery, Bank Transfer, Credit Card), and order confirmation — for both guest and authenticated users.
  • User Retention: validate the post-purchase experience: invoice listing, INV-prefixed references, invoice detail pages, and the "Download PDF" journey that supports customer self-service.
  • Data & Access Security: verify two-role authentication (Customer, Admin), role separation, invalid-credential rejection with clear error messages, and that password inputs mask their content.

2.2 Quality-Engineering Objectives

  • Zero-Flakiness Automation: eliminate artificial failures using Playwright's auto-waiting locators, response-driven waits (waitForResponse), and a hardened multi-role authentication setup.
  • High Functional Coverage: cover every in-scope module with happy-path and negative/edge-case scenarios, driven by externalised JSON data only — no hard-coded inputs inside spec files.
  • Network Resilience: intercept and mock backend responses (HTTP 500, 401, empty payload, network abort) to prove the Angular front-end degrades gracefully rather than crashing.
  • Quality Gates: integrate WCAG 2.0 A/AA accessibility auditing and visual layout-integrity checkpoints into every CI run.
  • Pipeline Readiness: execute in parallel across three CI shards; capture screenshots/video/traces on every failure boundary; publish a merged Allure dashboard to GitHub Pages automatically on each push to main.

6. Core Playwright Concepts — Auto-Waiting

Auto-waiting is the most impactful Playwright feature. It eliminates the need for artificial sleeps or manual WebDriverWait loops that make Selenium suites brittle.

6.1 What Auto-Waiting Checks

Before executing any action (click, fill, selectOption), Playwright polls the element until all of the following conditions are satisfied simultaneously:

Attached

The element is present in the DOM and not dynamically removed by the Angular change-detection cycle.

Visible

Has a non-zero bounding box and non-zero opacity; not hidden by display:none or visibility:hidden.

Stable

Its position does not change between two consecutive animation frames — prevents clicking on an element mid-transition.

Enabled

Is not disabled (for <button>, <input>, <select> controls).

6.2 Why This Matters for Angular SPAs

Angular's change detection renders the DOM asynchronously in response to HTTP responses, user events, and zone-tracked timers. A product grid that was empty on page load may have 30 cards 400ms later after the catalog API responds. Without auto-waiting, a driver.findElement() call at the wrong millisecond throws StaleElementReferenceException. Playwright simply retries internally until the element is ready.

6.3 Response-Driven Waits (Beyond Auto-Wait)

For actions that trigger an API call before the DOM updates, the framework registers a response listener before the action and awaits it afterwards — creating a zero-artificial-delay synchronisation point:

// Wait for the product search API before asserting results
const searchReady = page.waitForResponse(
  r => r.url().includes('/products/search') && r.status() === 200
);
await homePage.searchInput.fill('Pliers');
await homePage.searchButton.click();
await searchReady;   // resolves the instant the API responds — no sleep needed
expect(await homePage.productNames.count()).toBeGreaterThan(0);
Comparison with Selenium: In Selenium, the equivalent requires a FluentWait<WebDriver> with polling and a custom ExpectedCondition — tens of lines of boilerplate. In Playwright, auto-waiting is built into every action call.

7. Core Playwright Concepts — Locator Strategies

Choosing the right locator determines whether a test survives a UI redesign. The framework uses a priority-ordered strategy.

7.1 Priority Ordering

PriorityLocator APIExample from codebaseWhy preferred
1 (best)getByTestId()page.getByTestId('nav-menu')An explicit test contract between developer and QA. Survives copy-writing, CSS-class changes, and layout overhauls.
2getByRole()page.getByRole('button', {name:'Update Profile'})Reflects semantic accessibility intent. Resilient to class and attribute changes.
3getByLabel()page.getByLabel('Hand Tools')Links a form control to its label — robust even when the input's id is dynamically generated.
4getByText() / regexpage.getByText(/INV-/)When no test ID exists. A regex avoids brittle exact-text matching.
5locator() CSSpage.locator('.alert-success')Last resort — used only when no semantic hook is available on the element.

7.2 Global testIdAttribute Configuration

The application's test-ID attribute is data-test (e.g., <input data-test="email">). Playwright's global config maps getByTestId() to this attribute, so no spec file needs to know the raw selector:

// playwright.config.js — one-line global mapping
use: { testIdAttribute: 'data-test' }

// Then in any spec file — no raw selector visible:
await page.getByTestId('email').fill(email);       // ⟶ [data-test="email"]
await page.getByTestId('login-submit').click();    // ⟶ [data-test="login-submit"]

7.3 Dynamic / Partial Locators

Admin CRUD buttons (e.g., brand-123-edit) have dynamically generated IDs that change with each record. The framework uses attribute-prefix matching to locate them regardless of the ID suffix:

// Matches ANY element whose data-test starts with "brand-" and ends with "-edit"
this.editBrandBtn = page.locator('[data-test^="brand-"][data-test$="-edit"]').first();

7.4 Locator Chaining and Filtering

// Filter rows by cell content (used to find the INV- reference on the invoice list)
this.firstInvoiceListCell = page.locator('td').filter({ hasText: /INV-/ }).first();

// Chain: find a row whose first-name cell matches, then target a button within it
const row = page.locator('table tbody tr').filter({ hasText: 'Krishna' });
await row.getByRole('button', { name: 'Edit' }).click();

8. Core Playwright Concepts — Assertions

8.1 Web-First Assertions

Playwright's expect(locator).* assertions are web-first: they automatically retry the check until it passes or the timeout expires. This prevents the "stale snapshot" failure pattern where a single synchronous comparison catches the DOM in a transitional state.

// Retries until nav-menu is visible (up to actionTimeout = 15 s)
await expect(page.getByTestId('nav-menu')).toBeVisible();

// Retries until URL matches the pattern
await expect(page).toHaveURL(/.*account/);

// Retries until the input holds the expected value
await expect(profilePage.firstNameInput).toHaveValue('Krishna');

// Retries until the invoice readonly input contains INV-
await expect(accountPage.invoiceIdCell).toHaveValue(/INV-/);

8.2 Soft Assertions

Standard assertions abort the test on first failure. Soft assertions (expect.soft()) continue execution and collect all failures together, reporting them at the end. They are used for non-critical metadata checks where seeing all failures at once is more useful than stopping at the first one:

// Both checks run even if the first one fails — all failures reported together
await expect.soft(page).toHaveURL(/.*login/);
await expect.soft(
  page.locator('[data-test$="-error"]').first()
).toBeVisible();

8.3 Negative Assertions

// Assert a button is disabled (checkout validation)
await expect(checkoutPage.finishButton).toBeDisabled();

// Assert a field is not empty (profile pre-populated)
await expect(profilePage.firstNameInput).not.toBeEmpty({ timeout: 10000 });
Key principle: assertions live only in spec files. Page Object classes expose locators; they never assert. This separation means changing an assertion (e.g., switching from URL pattern to page-title check) requires editing one spec line, not searching the entire POM class tree.

8.4 Input Value vs Text Content

A common mistake is using toContainText() on a form input. Inputs do not have text content — they have a value attribute. The framework uses the correct assertion for each element type:

// ✅ Correct — for <input> / <textarea> / <select>
await expect(accountPage.invoiceIdCell).toHaveValue(/INV-/);

// ✅ Correct — for visible text in <p> / <td> / <h1> etc.
await expect(adminPage.pageTitle).toContainText('Sales over the years');

9. Framework Design Pattern — Page Object Model

The Page Object Model (POM) is the structural backbone of the framework. Its core rule: every selector lives in a page class; every assertion lives in a spec.

9.1 Class Hierarchy

9.2 A Complete Page Object (AdminPage excerpt)

export class AdminPage extends BasePage {
  constructor(page) {
    super(page);
    // Navigation links (data-test attributes)
    this.navMenu       = page.getByTestId('nav-menu');
    this.brandsLink    = page.getByTestId('nav-admin-brands');

If the application renames data-test="nav-admin-brands" to data-test="admin-nav-brands", only one line in AdminPage.js changes — not the 16 admin tests that reference adminPage.brandsLink. Without POM, each of those 16 tests would need an independent update.

10. Framework Design Pattern — Fixtures & Dependency Injection

10.1 What Are Playwright Fixtures?

Playwright's fixture system is the DI container of the framework. Each fixture is a function that receives its own dependencies and calls use(value) to provide the value to a test. The runner manages the lifecycle — setup before the test, teardown after — automatically.

10.2 The Custom Fixture Layer

// fixtures/index.js — extends the base test with one fixture per page object
import { test as baseTest } from '@playwright/test';
import { LoginPage }    from '../pages/LoginPage.js';
import { CheckoutPage } from '../pages/CheckoutPage.js';
// … all other imports

export const test = baseTest.extend({
  loginPage:    async ({ page }, use) => await use(new LoginPage(page)),
  checkoutPage: async ({ page }, use) => await use(new CheckoutPage(page)),
  // … all others follow the same pattern
});

export { expect } from '@playwright/test';

10.3 Usage in Spec Files

Spec files import from fixtures/index.js — not from @playwright/test directly — and declare only the objects they need. The runner instantiates exactly those objects:

import { test, expect } from '../fixtures/index.js';

// Checkout spec: needs three fixtures; receives them pre-constructed
test('TC10_Guest - Complete E2E Checkout',
  async ({ homePage, productPage, checkoutPage }) => {
    await homePage.goTo();
    await homePage.productNames.first().click();
    await productPage.clickAddToCart();
    await checkoutPage.continueAsGuest();
    // … assertion
  }
);
Benefit: A spec that does not need adminPage will never instantiate it. This keeps each test's setup cost proportional to what it actually uses — no "god object" passed to every test.

12. Authentication Strategy & storageState

12.1 The Problem with Per-Test Login

A naïve approach performs a full UI login in every test's beforeEach. For 139 tests this means 139 login flows — each taking 2–5 s and creating an additional Cloudflare-challenge opportunity. This adds up to 5–11 minutes of pure overhead, plus significantly higher flakiness risk from repeated credential submissions to the demo site.

12.2 storageState — Session Capture

Playwright's page.context().storageState({ path }) serialises the browser's cookies, localStorage, and sessionStorage to a JSON file. A dedicated setup project (auth.setup.js) runs once before all functional tests and saves both role sessions:

// auth.setup.js
setup('Authenticate Customer', async ({ page }) => {
  await gotoLogin(page);                   // navigates + waits for form
  await page.getByTestId('email').fill(email);
  await page.getByTestId('password').fill(password);
  await page.getByTestId('login-submit').click();
  // 30 s — 3 shards log in simultaneously; the demo's API can lag
  await expect(page.getByTestId('nav-menu')).toBeVisible({ timeout: 30000 });
  await page.context().storageState({ path: 'playwright/.auth/user.json' });
});

12.3 Session Replay — Zero Login Cost per Test

Functional tests declare dependencies: ['setup'] and load the saved session automatically:

// playwright.config.js
projects: [
  { name: 'setup', testMatch: /.*\.setup\.js/ },
  {
    name: 'chromium',
    use: { storageState: 'playwright/.auth/user.json' },
    dependencies: ['setup'],  // always runs setup first
  },
]

13. Network Interception — page.route() Deep Dive

13.1 How page.route() Works

page.route(pattern, handler) registers an intercept handler that runs before any matching network request leaves the browser. The handler receives a route object and must call exactly one of three methods to resolve it:

route.fulfill(opts)

Return a synthetic response. The real server is never contacted. Set any status, headers, and body.

route.continue()

Let the real request proceed normally. Useful for observing that a call was made without altering its response.

route.abort()

Simulate a network error (connection refused). The browser sees a failed request — no server involvement.

13.2 Pattern Matching

// Regex matching the catalog API on any path
const PRODUCTS_API = /api\.practicesoftwaretesting\.com\/products/;

// Register BEFORE navigation so the handler is in place when the page loads
await page.route(PRODUCTS_API, route =>
  route.fulfill({
    status: 500,
    contentType: 'application/json',
    body: JSON.stringify({ message: 'Internal Server Error' }),
  })
);
await homePage.goTo();
await expect(homePage.searchInput).toBeVisible({ timeout: 15000 }); // shell renders
expect(await homePage.productNames.count()).toBe(0);                  // no products

13.3 Observation Pattern (continue)

let intercepted = false;
await page.route(PRODUCTS_API, route => {
  intercepted = true;
  return route.continue();     // real server still responds
});
await homePage.goTo();
await homePage.productNames.first().waitFor({ state: 'visible', timeout: 15000 });
expect(intercepted).toBe(true);   // confirms the API call was made
Why this matters in production QA: Network mocking proves that the UI layer has a clean separation from the API layer. If a 500 response white-screens the app, automation catches it in seconds on the next CI push, before any user encounters it.

14. Parallel Execution, Sharding & CI/CD Pipeline

14.1 Workers — Within a Single Machine

Setting fullyParallel: true and workers: 2 allows Playwright to run two tests simultaneously in separate browser contexts on one runner. Each context is completely isolated (separate cookies, tabs, and network state). This doubles throughput while keeping request concurrency low enough to avoid additional Cloudflare scrutiny on the shared demo.

14.2 Sharding — Across Multiple Machines

The GitHub Actions matrix creates three independent runners. Each runner receives a --shard=N/3 flag that tells Playwright to execute only its assigned slice of tests:

# .github/workflows/playwright.yml
strategy:
  fail-fast: false       # keep the other shards running if one fails
  matrix:
    shard: [1, 2, 3]

- name: Run Playwright Tests (shard ${{ matrix.shard }}/3)
  run: xvfb-run -a npm run test:ci -- --project=chromium --shard=${{ matrix.shard }}/3

Combining workers and shards: 3 machines × 2 workers = 6 concurrent browser contexts. Wall-clock dropped from 47 minutes (serial) to approximately 12 minutes — well inside the demo's auth-session window.

14.3 xvfb — Headed Chrome on a Headless Linux Server

GitHub-hosted Linux runners have no physical display. xvfb-run -a creates a virtual framebuffer (X Virtual Frame Buffer) so Chrome can launch in headed mode. Headed mode is required because Cloudflare's JavaScript challenge runs in the browser's rendering engine and cannot complete in headless mode (which exposes navigator.webdriver = true).

14.4 Allure Artifact Merging

# Each shard uploads under a unique name
- uses: actions/upload-artifact@v4
  with: { name: allure-results-${{ matrix.shard }}, path: project1-pom/allure-results }

# The report job downloads and merges all three
- uses: actions/download-artifact@v4
  with:
    pattern: allure-results-*
    path: project1-pom/allure-results
    merge-multiple: true        # flattens into one directory

14.5 Full Pipeline Flow

On every push to any branch:

  1. test × 3 (parallel): each shard installs deps, installs Chrome, runs auth.setup.js, executes its slice of 139 tests, uploads Allure results.
  2. report (sequential, after all shards): downloads and merges all three result sets, generates the Allure HTML report, deploys to GitHub Pages.

The Pages deploy only fires on pushes to main or master (configurable), so branch PRs get the artifact-only report while main publishes publicly.

19. Defects & Bugs Discovered

Automation revealed two genuine application-level defects in the Toolshop platform. Because the AUT is a third-party demo, these cannot be patched. They are documented as explicit tests that assert the current (buggy) behaviour and are annotated to be flipped once the application is corrected.

IDDescriptionSpec / TCSeverity
BUG-01 Alphabetic input bypasses postal-code and house-number validation. The address form accepts alphabetic characters in numeric-only fields and does not disable the "Proceed" button. Expected: validation error + button disabled. Actual: button enabled, checkout proceeds. 05b TC24_Auth_Neg Medium
BUG-02 Finish button enabled without an account name. With bank name and account number filled but account name empty (a required field), the Finish button remains enabled. Expected: disabled until all required bank fields are complete. 10 TC21_Auth_Neg Medium
Documentation approach: Instead of skipping or marking xfail, the assertions are inverted to match the actual buggy behaviour (e.g., toBeEnabled() instead of toBeDisabled()). This keeps the tests green and provides an instant regression signal — when the app is fixed, the test will fail and alert the team to re-verify and correct the assertion.

20. Challenges & Resolutions

20.1 Cloudflare Bot Protection

Problem: The public AUT sits behind Cloudflare. The initial self-hosted Windows runner had a single static IP that was flagged as a bot after repeated automated login attempts. Once flagged, every login returned a challenge interstitial instead of the Angular login form — getByTestId('email') never appeared, auth.setup.js timed out, and the entire 84-test dependent suite was skipped at once.

Root cause: Headless Chrome (the default when CI=true) sets navigator.webdriver = true, a fingerprint Cloudflare uses to identify bots. Self-hosted runners reuse the same IP, concentrating challenge risk.

Resolution: Moved to GitHub-hosted ubuntu-latest. Run Chrome headed via xvfb-run with --disable-blink-features=AutomationControlled. GitHub's rotating runner IPs and headed real Chrome allow the Cloudflare JS challenge to auto-solve in the browser's rendering engine.

20.2 Authentication Session Expiry (47-Minute Serial Run)

Problem: The demo issues short-lived JWT tokens. A 47-minute serial run outlived the session captured at setup; login-dependent tests in the second half of the suite all failed simultaneously with "nav-menu not visible."

Resolution: Three-shard × two-worker parallelism cut wall-clock to ~12 minutes. Each shard runs auth.setup.js independently, ensuring a fresh valid session throughout.

20.3 Wrong DOM Assumptions

Problem: Several test assertions were based on inferred DOM structure rather than observed evidence — the invoice reference was in an <input> value (not text content); the site has no <footer> tag; the contact form does not auto-fill from the session.

Resolution: Allure's failure ARIA snapshots (error-context.md) showed the exact DOM at failure time. Each locator was corrected to match reality.

20.4 Transient Shard Skips on Login

Problem: With 3 shards authenticating simultaneously, the demo's login API occasionally lagged beyond the original 15-second post-login wait, causing one shard's setup to time out and its entire test set to be dependency-skipped.

Resolution: Hardened the post-login visibility wait from 15s to 30s. The broader window absorbs burst-login lag without impacting normal-run speed.

20.5 Visual Baseline Portability

Problem: toHaveScreenshot() requires OS/renderer-specific committed PNG baselines; a baseline from Windows 120dpi fails on Linux CI 96dpi virtual framebuffer even for identical content.

Resolution: Adopted layout-integrity checkpoints (bounding-box assertions + PNG attachment) — deterministic across environments. The pixel-diff upgrade path is documented for use in a controlled environment.
