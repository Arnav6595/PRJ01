1. Overview

The Application Under Test (AUT) is “Practice Software Testing – Toolshop” (practicesoftwaretesting.com), a fully functional, Angular-based e-commerce platform that simulates real-world retail operations. The system supports a complete user journey: product discovery, search and filtering, shopping-cart management, multi-step checkout with several payment methods, customer support, invoice/order history, and a privileged administrative dashboard.

This document presents the quality-engineering strategy and the implementation of a production-ready Playwright automation framework built from scratch in JavaScript. The framework isolates front-end UI testing from a volatile, third-party backend, handles a single-page Angular application reliably with auto-waiting locators, and runs unattended inside a Continuous Integration pipeline that publishes a rich Allure execution dashboard.

The engagement followed the full Software Testing Life Cycle (STLC): requirement analysis, test strategy and design, framework construction, execution, defect documentation, and reporting. The final suite comprises 139 automated tests — 124 functional tests across eight services plus 15 advanced-verification tests (network mocking, accessibility, and visual checkpoints) — executing in parallel across a three-way sharded CI matrix at a 100% pass rate.

2. Goals

The primary quality objectives for this engagement are defined below.

2.1 Business Objectives

  • Transaction Integrity: guarantee that the end-to-end checkout flow — address entry, payment-method selection, and order confirmation — behaves correctly for both guest and authenticated users.
  • User Retention: validate the post-purchase experience: invoice listing, invoice detail pages, and the “Download PDF” journey that supports customer self-service.
  • Data & Access Security: verify authentication, role separation (Customer vs Admin), invalid-credential handling, and that the password field is masked.

2.2 Quality-Engineering Objectives

  • Zero-Flakiness Automation: eliminate artificial failures using Playwright’s auto-waiting locators, response-driven waits, and a hardened authentication setup.
  • High Functional Coverage: cover every in-scope module with both happy-path and negative/edge-case scenarios, driven by externalised data.
  • Network Resilience: intercept and mock backend responses (HTTP 500/401, empty, aborted) to prove the front-end degrades gracefully.
  • Quality Gates: integrate automated accessibility auditing (axe-core, WCAG 2 A/AA) and visual UI checkpoints into the regression run.
  • Pipeline Readiness: run the suite headlessly-equivalent in CI, in parallel across shards, with screenshots/video/traces captured on failure and an Allure dashboard published to GitHub Pages.

  • Accessibility auditing — axe-core WCAG scans as a monitoring layer.
  • Visual checkpoints — widget capture with layout-integrity assertions.

2.3 Page Object Model (POM)

Every page is a class extending BasePage, exposing locators and high-level actions while keeping selectors entirely separated from test assertions — satisfying the “strict POM” requirement. Locators prefer the application’s data-test attributes (configured globally via testIdAttribute: 'data-test'), with role/text locators as resilient fallbacks.

// pages/LoginPage.js (excerpt)
export class LoginPage extends BasePage {
  constructor(page){
    super(page);
    this.emailInput   = page.getByTestId('email');
    this.passwordInput= page.getByTestId('password');
    this.loginButton  = page.getByTestId('login-submit');
  }
  async login(email, password){
    await this.fillInput(this.emailInput, email);
    await this.fillInput(this.passwordInput, password);
    await this.clickElement(this.loginButton);
  }
}

2.4 Fixtures (Dependency Injection)

A custom test fixture instantiates each page object and injects it, so specs request only what they need (async ({ homePage, checkoutPage }) => …). This removes boilerplate and centralises object lifecycle.

Milestone 3 — Functional Test Suites

3.1 Design Methodology

Tests are built around three principles: path verification (following the critical user journey end-to-end), assertion-driven validation (every test asserts an explicit expected state via Playwright web-first assertions), and requirement traceability (each test maps to an FR from Milestone 1). The suite contains 124 functional tests (including the two authentication-setup tests) across the eight services. An additional 40 functional tests were authored to deepen coverage — five per service — each placed beneath a clearly marked NEW TEST CASES (added 2026-06-01) boundary comment in its spec file.

Milestone 5 — CI/CD Pipeline, Cloudflare Resolution & Allure Reporting

5.1 GitHub Actions Workflow

The pipeline (.github/workflows/playwright.yml) installs dependencies, caches npm, installs the Chrome browser, executes the sharded suite, uploads Allure raw results per shard, then a dedicated report job merges them and publishes the HTML Allure report to GitHub Pages.

# test job — 3-way matrix
strategy:
  fail-fast: false
  matrix:
    shard: [1, 2, 3]
steps:
  - run: npm ci
  - run: npx playwright install chrome --with-deps
  - run: xvfb-run -a npm run test:ci -- --project=chromium --shard=${{ matrix.shard }}/3
  - uses: actions/upload-artifact@v4
    with: { name: allure-results-${{ matrix.shard }}, path: project1-pom/allure-results }

5.2 The Cloudflare Challenge & Its Resolution

The most significant engineering challenge was that the public AUT sits behind Cloudflare bot protection. The resolution was empirical and iterative:

  1. Self-hosted runner (blocked): a single static IP was repeatedly challenged; once flagged, every login failed and the whole dependent suite collapsed.
  2. GitHub-hosted runner + headed Chrome: moving to ubuntu-latest and running real Chrome headed via xvfb (with the automation fingerprint reduced) allowed the Cloudflare JavaScript challenge to auto-solve. This bypassed the block.
  3. Session expiry (root-caused): the demo issues short-lived auth tokens and resets data periodically; a long serial run outlived the session captured at setup, so late login-dependent specs failed. Sharding (3 machines, fresh login each) + parallelism brought the suite well within the session window.
  4. Transient stabilisation: reduced per-shard workers, increased retries, and hardened the authentication wait (15s → 30s) to remove intermittent shard skips.

5.3 Sharding & Parallelism

The suite is split across three parallel runners via --shard=N/3; each shard authenticates independently and runs roughly a third of the tests. Per-shard Allure artifacts are merged with download-artifact (pattern: allure-results-*, merge-multiple: true) into a single combined report.

5.4 Allure Reporting — Module-Wise

11. Conclusion

This capstone delivers a complete, production-ready Playwright automation framework for the Toolshop e-commerce platform. It implements a strict Page Object Model with dependency-injected fixtures, fully externalised data, a multi-role authentication matrix, and failure-boundary diagnostics. Beyond functional coverage of all eight services, it adds three advanced verification layers — network interception/mocking, accessibility auditing, and visual checkpoints — and runs unattended in a sharded GitHub Actions pipeline that publishes an Allure dashboard.

Critically, the project overcame a real-world Cloudflare bot-protection barrier and a short-lived-session constraint through an evidence-driven sequence of infrastructure and design changes, arriving at a stable 139-test suite at a 100% pass rate. The framework is maintainable, CI-ready, and structured to extend cleanly as the application evolves.
